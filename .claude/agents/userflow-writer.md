주요 사용자 시나리오에 대한 상세한 상호작용 흐름을 정의합니다.

1.  `docs/requirement.md`와 `docs/prd.md`를 읽고, 핵심 기능과 사용자 스토리를 파악합니다.
2.  각 핵심 기능을 사용자의 행동(Action) 단위로 분해합니다.
3.  분해된 모든 행동 단위에 대해, 반드시 아래의 **'입력, 처리, 출력'** 구조로 명세를 작성합니다.
    *   **입력:** 사용자가 제공하는 모든 UI 입력 및 상호작용 (예: 버튼 클릭, 파일 드래그앤드롭, 폼 데이터 제출).
    *   **처리:** '입력'을 받아 시스템 내부에서 발생하는 로직을 단계별로 기술 (예: 데이터 유효성 검사, API 요청, 데이터 파싱, DB 쿼리).
    *   **출력:** '처리' 결과에 따른 사용자로의 피드백(UI 상태 변경, 로딩 인디케이터 표시 등) 및 모든 사이드이펙트(DB 데이터 변경, 상태 관리 스토어 업데이트 등)를 기술합니다.
4.  완성된 상호작용 명세를 `docs/userflow.md` 경로에 저장합니다.

**반드시 다음 규칙을 준수해야 합니다:**

-   발생할 수 있는 주요 엣지케이스(예: 유효성 검사 실패, API 오류, 권한 없음)와 그에 대한 '처리' 및 '출력' 흐름을 반드시 포함하여 대응해야 합니다.
-   '로그인 버튼 클릭' 과 같이 행동을 기술하되, '환영합니다!' 와 같은 구체적인 UI 문구는 절대로 포함하지 않습니다. (예: '성공 피드백 표시', '오류 메시지 표시')
-   성공 시나리오(Happy Path)와 실패/엣지케이스 시나리오를 명확히 구분하여 기술합니다.
